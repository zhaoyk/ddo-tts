# MiMo V2.5 TTS 支持设计

- 日期：2026-07-10
- 状态：设计已在对话中逐节批准，书面规格待用户复核
- 目标仓库：`ddo-tts`
- 官方接口文档：[语音合成（MiMo-V2.5-TTS 系列）](https://mimo.mi.com/docs/zh-CN/quick-start/usage-guide/audio/speech-synthesis-v2.5)

## 1. 背景

`ddo-tts` 的章节朗读由 `ReadAloud` 选择一个 `BaseReadAloudService` 子类，再由具体 Service 完成合成、播放、段落进度和章节切换。项目已有系统 TTS、HTTP TTS、Edge 和豆包实现。

本功能新增一个内置 MiMo 云端朗读引擎。它使用用户自己的 MiMo API Key，通过 `mimo-v2.5-tts` 的预置音色生成 WAV 音频，并沿用当前阅读进度、通知控制以及“朗读与手动翻页解耦”行为。

## 2. 目标

1. 在朗读引擎列表中提供原生“MiMo TTS”选项，可设为全局或单本书引擎。
2. 使用用户本地配置的 API Key 和官方全部预置音色调用 `mimo-v2.5-tts`。
3. 支持全局朗读风格，并允许当前书籍完全覆盖全局风格。
4. 每个原始段落独立合成，保持段落跳转和进度映射准确。
5. 当前段可播放后，继续串行合成本章剩余段落，并预取下一章前 10 段。
6. 音频仅缓存在内存中；播放倍速由 ExoPlayer 本地控制，不因调速重复调用 API。
7. 对认证、请求参数、限流、网络和响应格式错误给出明确且不泄露密钥的反馈。
8. MiMo 实现与 Edge、豆包、HTTP TTS 相互隔离，避免改变现有引擎行为。

## 3. 非目标

本次不支持以下能力：

- `mimo-v2.5-tts-voicedesign` 文本设计音色；
- `mimo-v2.5-tts-voiceclone` 音色复刻；
- PCM16 真流式边接收边播放；
- 自定义 Base URL、代理地址或区域选择；
- 中英文段落自动检测并切换音色；
- 设置页试听或独立 API Key 验证；
- 划词朗读 `help/TTS.kt` 使用 MiMo；
- API Key 加密存储；
- MiMo 失败时自动回退到其他 TTS；
- 对 Edge、豆包或 HTTP TTS 做公共基类重构。

## 4. 已确认的产品决策

| 主题 | 决策 |
|---|---|
| 模型范围 | 仅 `mimo-v2.5-tts` 预置音色 |
| API Key | 用户填写，明文保存到应用私有 `TTS_CONFIG` |
| API 地址 | 固定 `https://api.xiaomimimo.com/v1/chat/completions` |
| 输出方式 | 非流式 WAV，整段生成完成后播放 |
| 分段 | 每个原始段落单独请求 |
| 预取 | 串行合成本章剩余段落，再预取下一章前 10 段 |
| 缓存 | 仅内存缓存 |
| 音色 | 全局配置，提供全部官方预置音色 |
| 风格 | 全局配置；单本书可以完全覆盖 |
| 语速 | ExoPlayer 本地倍速，不重新合成 |
| 失败行为 | 暂停并保留位置，不自动回退 |
| 试听 | 不提供 |

## 5. 架构

### 5.1 引擎路由

在 `ReadAloud.getReadAloudClass()` 中识别引擎值 `mimo`，返回 `TTSMiMoAloudService`。MiMo Service 在 Manifest 中注册为 `mediaPlayback` 前台服务。

引擎仍使用现有 JSON `SelectItem` 格式保存，例如：

```json
{"title":"MiMo TTS","value":"mimo"}
```

MiMo 不改变其他引擎的判断顺序或默认回退逻辑。

### 5.2 组件职责

#### `TTSMiMoAloudService`

继承 `BaseReadAloudService`，负责：

- 将当前阅读章节映射为待合成段落；
- 编排当前段、本章剩余段落和下一章前 10 段的串行合成；
- 管理合成任务、OkHttp Call 的取消和 Service 生命周期；
- 维护 WAV 内存缓存及 ExoPlayer 播放列表；
- 处理播放、暂停、恢复、跳段、切章、停止和倍速；
- 根据播放进度更新朗读高亮；
- 将客户端错误转换为用户提示和暂停状态。

它不构造 HTTP JSON，也不直接读取或修改设置界面控件。

#### `MiMoTtsClient`

负责：

- 构造官方 v2.5 非流式请求；
- 设置 `api-key` 和 JSON Header；
- 执行可取消的 OkHttp 请求；
- 解析成功响应并解码 Base64 WAV；
- 解析 MiMo 错误对象、HTTP 状态和请求 ID；
- 返回音频字节或结构化 `MiMoTtsError`。

客户端不依赖 `ReadBook`、Service、Toast 或 ExoPlayer。生产环境固定使用官方 URL；测试可以注入本地 URL 和测试 OkHttpClient。

#### `MiMoTtsConfig`

集中解析最终生效配置：

- 全局 API Key；
- 全局音色；
- 全局风格；
- 当前书籍的可选风格覆盖。

Service 只接收最终不可变配置快照，避免在一次段落合成过程中读到互相不一致的字段。

#### `MiMoMemoryDataSource`

把内存中的 WAV `ByteArray` 提供给 Media3。它使用独立名称和实现，不依赖 Edge 文件中的 `ByteArrayDataSource`，从而避免 MiMo 与 Edge 的隐式耦合。

#### `MiMoTtsConfigDialog`

负责编辑和持久化配置，不发起网络请求。`SpeakEngineDialog` 只负责显示 MiMo 引擎项、选择引擎和打开此对话框。

## 6. 配置与持久化

### 6.1 全局配置

使用应用私有的 `SharedPreferences("TTS_CONFIG", MODE_PRIVATE)`：

| Key | 类型 | 默认值 | 含义 |
|---|---|---|---|
| `tts_mimo_api_key` | String | 空 | 用户的 MiMo API Key |
| `tts_mimo_voice` | String | `mimo_default` | 全局预置音色 |
| `tts_mimo_style` | String | 空 | 可选的全局自然语言朗读风格 |

API Key 按已批准要求明文保存，但输入框默认遮罩。Key 不进入日志、Toast、异常文本或缓存键。

### 6.2 单本书配置

在 `Book.ReadConfig` 中增加：

```text
mimoTtsStyle: String? = null
```

`Book.ReadConfig` 已作为 JSON 写入现有 Room 列；新增可空字段不改变表结构，因此不增加数据库版本或 Room migration。旧 JSON 缺少该字段时，Gson 将其解析为 `null`。

最终风格规则：

1. 单书 `mimoTtsStyle` 去除首尾空白后非空：完全覆盖全局风格；
2. 单书字段为空或 `null`：继承全局风格；
3. 最终风格为空：请求中省略 `user` 消息。

全局风格与单书风格不拼接，避免出现矛盾指令和难以解释的优先级。

### 6.3 音色列表

设置页展示以下官方预置音色，并标注语言与性别：

- MiMo 默认：`mimo_default`；
- 冰糖：中文、女性；
- 茉莉：中文、女性；
- 苏打：中文、男性；
- 白桦：中文、男性；
- Mia：英文、女性；
- Chloe：英文、女性；
- Milo：英文、男性；
- Dean：英文、男性。

音色始终为全局配置，不提供单书覆盖或自动语言切换。

### 6.4 设置界面

MiMo 专用设置对话框包含：

- API Key 密码输入框及显示/隐藏开关；
- 全部预置音色选择器；
- 可选的全局风格多行输入框；
- 当前打开书籍存在时显示的单书风格多行输入框；
- “清除本书设置”操作；
- 保存与取消操作。

界面明确说明风格是不会被朗读的表演指导。保存时只做去除首尾空白和音色白名单校验，不调用 MiMo。API Key 可以先保存为空；真正选择 MiMo 开始朗读时给出缺少 Key 的可操作提示。

修改 API Key、音色或当前生效风格后，运行中的 MiMo Service 清除错误状态和不兼容缓存，从当前段重新建立播放队列。

## 7. MiMo API 协议

### 7.1 请求

```http
POST https://api.xiaomimimo.com/v1/chat/completions
api-key: <用户配置>
Content-Type: application/json
```

有风格时：

```json
{
  "model": "mimo-v2.5-tts",
  "messages": [
    {"role": "user", "content": "平静自然地讲述小说，情绪克制。"},
    {"role": "assistant", "content": "需要朗读的正文段落"}
  ],
  "audio": {
    "format": "wav",
    "voice": "mimo_default"
  }
}
```

无风格时，`messages` 只包含 `assistant` 正文。请求不设置 `stream: true`。

正文继续使用现有 `AppPattern.notReadAloudRegex` 清理逻辑。清理后为空的段落直接使用应用内静音 WAV，不调用 MiMo。

### 7.2 成功响应

客户端从以下路径读取音频：

```text
choices[0].message.audio.data
```

该值必须是非空 Base64。解码后必须得到非空字节并具有基本的 RIFF/WAVE 文件标识；随后由 Media3 完成实际播放格式验证。缺少字段、非法 Base64、空音频或缺少 WAV 标识均转换为响应格式错误。

### 7.3 敏感信息边界

禁止记录：

- `api-key` Header；
- 完整请求 Header；
- 完整请求 JSON；
- 正文和风格原文；
- Base64 音频；
- 包含上述内容的异常对象字符串。

允许记录经过脱敏的 HTTP 状态、MiMo 错误码、服务端请求 ID、段落索引、重试次数和耗时。

## 8. 合成、预取与缓存

### 8.1 段落粒度

每个 `BaseReadAloudService.contentList` 原始段落对应一次 MiMo 请求和一个播放项。这样保留上一段、下一段、章节位置和高亮的现有语义。

### 8.2 调度顺序

1. 优先合成当前段；
2. 当前段完成后立即加入 ExoPlayer 并允许开始播放；
3. 后台按段落顺序串行合成本章全部剩余段落，每完成一段就追加到播放列表；
4. 本章剩余段落全部处理后，串行预取下一章前 10 个非空段落；
5. 进入下一章时复用已预取字节，并继续合成该章其余段落；
6. 任一时刻不并发发起多个 MiMo 合成请求。

暂停只暂停 ExoPlayer，不取消上述串行合成计划；本章剩余段落和下一章前 10 段仍继续合成。停止、跳段、显式切章、切换引擎或修改生效配置会取消失效 Call 和预取任务。

### 8.3 内存缓存

缓存只存在于 MiMo Service 生命周期内，不写磁盘。合成签名包含：

```text
model + voice + effectiveStyle + normalizedText
```

缓存键由合成签名计算，不包含 API Key 或播放倍速。API Key 变化时仍主动清空缓存，避免跨账号继续使用旧请求结果。

播放项使用独立段落标识，包含章节和段落位置，避免相同正文产生的重复缓存键破坏播放进度。已播放且不会回退复用的音频及时释放；Service 销毁时清空全部字节。

本设计明确接受“整章剩余段落 + 下一章前 10 段”产生的 API 用量和 WAV 内存占用，不额外设置段数、字节数或章节长度上限。实现和用户文档需要保留这一风险说明。

## 9. 播放与进度

ExoPlayer 从 `MiMoMemoryDataSource` 读取完整 WAV。MediaItem 按章节和段落顺序加入播放列表，播放完成后沿用 `BaseReadAloudService` 的段落、页面和章节推进逻辑。

播放倍速计算沿用现有界面显示规则：

```text
playbackSpeed = (AppConfig.speechRatePlay + 5) / 10f
```

改变语速时调用 ExoPlayer 的播放倍速接口：

- 不停止当前段；
- 不清除播放列表；
- 不改变缓存键；
- 不重新调用 MiMo。

MiMo 的字级高亮仍为按音频时长和文本长度估算。墙钟延迟需要除以当前倍速，保证加速或减速后高亮不会系统性落后或超前。用户风格中描述的语速属于模型演绎，ExoPlayer 倍速是最终、确定性的二次调整。

手动翻页、去朗读页、从本页读和自动重新跟随继续使用现有 `ReadBook.aloudSyncView` 机制，不在 MiMo Service 中复制或改变该逻辑。

## 10. 错误处理与恢复

### 10.1 错误分类

| 条件 | 行为 |
|---|---|
| 未配置 API Key | 不请求；暂停并提示打开 MiMo 设置 |
| HTTP 400 | 参数/正文错误；不重复相同请求 |
| HTTP 401/403 | Key 无效或无权限；不重试，停止预取并暂停 |
| HTTP 429 | 有效 `Retry-After` 不超过 30 秒时按其等待；缺失时按退避策略重试；超过 30 秒时直接暂停并提示限流，最多重试两次 |
| 网络错误、连接/读取超时、HTTP 5xx | 间隔 1 秒、3 秒重试，最多两次 |
| 缺少音频、非法 Base64、空 WAV | 响应格式错误；重试一次 |
| 主动取消 | 静默结束，不显示失败 |

“最多重试两次”指首次请求失败后最多再发送两次，总尝试次数最多为三次。

### 10.2 超时

- 连接超时：30 秒；
- 单段完整响应上限：180 秒；
- 等待超过 10 秒：显示一次“MiMo 正在合成朗读音频”。

提示在单段请求期间只出现一次，不循环弹出。

### 10.3 前台段落与预取错误

当前必须播放的段落耗尽重试后：

- 暂停朗读；
- 保留当前章节、段落和播放位置；
- 给出认证、额度/限流、网络或响应格式对应提示；
- 用户修复配置或网络后点击继续，从缺失段重新请求。

后台预取遇到暂时性错误时：

- 不打断当前已经播放的音频；
- 停止继续向更远段落预取；
- 播放接近缺失段时执行一次前台恢复流程。

后台预取遇到 401/403 等确定性配置错误时立即暂停，因为继续使用相同配置请求没有意义。

不自动切换到系统 TTS、Edge、豆包或 HTTP TTS。

## 11. 生命周期与取消

| 事件 | MiMo 行为 |
|---|---|
| 播放 | 合成当前段，开始串行构建剩余播放列表 |
| 暂停 | 暂停 ExoPlayer；合成和预取继续 |
| 恢复 | 恢复 ExoPlayer；若此前因暂时错误停止预取则补齐 |
| 上一段/下一段 | 取消已失效请求，重建目标位置后的队列 |
| 自动进入下一章 | 复用下一章前 10 段缓存，继续合成其余段落 |
| 用户显式切章/从本页读 | 取消旧章节请求和队列，从新位置建立计划 |
| 修改 API Key/音色/生效风格 | 清缓存、清错误、从当前段重建 |
| 修改倍速 | 仅修改 ExoPlayer 倍速和高亮计时 |
| 停止或 Service 销毁 | 取消 Call/协程，释放 ExoPlayer 和全部缓存 |

- 手动翻页解耦行为无回归。

- 全部官方音色可选，API Key 默认遮罩；

## 13. 风险与取舍

1. **WAV 内存占用**：非流式 WAV 通常比 Edge MP3 更占内存，整章及下一章预取可能在长章节上产生显著峰值。用户已明确选择该策略，本次不增加上限。
2. **API 用量**：即使用户随后停止或跳章，已生成和已发送的预取请求可能仍产生用量。
3. **服务端风格不确定性**：自然语言风格不能保证每段完全一致；本地播放器倍速只保证最终速度倍率稳定。
4. **明文 Key**：应用私有 SharedPreferences 可防止普通应用直接读取，但不抵御已 root 设备、调试备份或设备被攻破。该风险为用户明确接受的产品决策。
5. **按段调用开销**：每个原始段落一次请求可保持控制准确，但比合并长文本有更多网络与推理开销。

## 14. 完成标准

当以下条件全部满足时，MiMo TTS 功能实现才视为完成：

- 本设计中所有目标能力均已实现；
